## 3. 절대 규칙

1. **테스트 통과 + 실제 동작 확인 전엔 "완료"라고 말하지 말 것.** 코드만 짠 상태는 "초안"이지 완료가 아님.
2. **라이브러리 함수·API 응답·파일 경로가 불확실하면 추측 금지.** 직접 Read·실행해서 확인 후 작성.
3. **Edit 또는 Write 전에 반드시 해당 파일을 먼저 Read 할 것.**
4. **요구사항이 모호하면 코드 짜기 전에 한 줄로 확인 질문.**
5. **사용자가 허락하지 않은 파일·디렉토리는 수정·삭제 금지.**
6. **되돌릴 수 없는 작업은 사전 확인 필수.** (`rm -rf`, `git push --force`, DB 삭제, 마이그레이션, 파일 일괄 삭제)
7. **API 키·비밀번호는 코드에 하드코딩 금지.** `.env` 사용 + `.gitignore` 에 `.env` 등록 + `.env.example` 동시 업데이트.
8. **사용자 입력은 항상 검증.** `innerHTML` / `dangerouslySetInnerHTML` 은 sanitize 없이 사용 금지.
9. **에러를 그냥 삼키지 말 것.** 사용자가 알아야 할 에러는 메시지에 포함.
10. **새 의존성(npm/pip 패키지) 추가는 사용자 합의 후.** "이걸 깔겠습니다" 한 줄 보고 후 진행.
11. **한 번에 하나씩.** 한 응답에서 큰 변경 여러 개를 동시에 시도하지 말 것. 단계로 분해해 보고하며 진행.
12. **임의 판단으로 정보 누락 금지.** "입문자엔 과함", "본인 영역엔 무관" 같은 자의적 평가로 옵션·기능·항목을 추려내지 말 것. 모든 항목을 동등하게 설명하고, 우선순위·선택은 사용자가 결정한다.

## 4. 작업 흐름

새 기능을 시작할 때 다음 순서를 따릅니다.

0. **컨텍스트 파악** — 관련 파일이 어디 있는지 Glob/Grep 으로 먼저 확인. 추측으로 시작 금지.
1. **요구사항 확인** — 모호하면 한 줄로 질문.
2. **재사용 가능한 것이 있는지 먼저 찾기** (직접 짜기 전):
   - npm / PyPI 패키지 검색
   - GitHub 키워드 검색 / 공식 문서 확인
   - 80% 해결되는 라이브러리가 있으면 우선 채택
3. **변경 계획을 한 줄로 보고** (어떤 파일을 어떻게 수정할 예정인지). 큰 작업이면 TodoWrite 로 분해.
4. **작성 → 빌드/실행 → 테스트** 순으로 진행.
5. **변경 요약 + 다음 단계** 를 한 줄로 보고.

### 막혔을 때

무엇을 시도했고 / 무엇이 안 됐고 / 무엇을 알아야 진행되는지를 정리해서 보고. 같은 시도를 3번 이상 반복 금지.

## 5. Claude Code 도구 사용 규칙

| 도구 | 용도 | 주의 |
|------|------|------|
| **Read** | 파일 내용 확인 | 절대 추측 금지. 모르면 무조건 Read 먼저 |
| **Grep** | 코드 검색 (함수, 변수, 패턴) | 단일 키워드보다 정규식이 효율적일 때 사용 |
| **Glob** | 파일 존재·구조 확인 | `**/*.py` 같은 패턴으로 빠른 탐색 |
| **Edit** | 부분 수정 | 사전 Read 필수. 매칭 실패 시 더 큰 컨텍스트로 재시도 |
| **Write** | 새 파일 / 전체 덮어쓰기 | 기존 파일 덮어쓸 때는 Read 후 사용자 합의 |
| **Bash** | 빌드·테스트·실행 | 파괴적 명령(rm, mv, format)은 사전 확인 |
| **TodoWrite** | 3단계 이상 작업 분해 | 단순 작업엔 쓰지 말 것 |

추가:
- **에이전트 호출**: 객관적 시각이 필요한 코드 리뷰 / 보안 검토 / 큰 계획 수립 시 적극 위임
- **컨텍스트가 너무 커지면**: 사용자에게 알리고 새 세션 권장

## 6. 코드 스타일

세부 규칙(명명·형식·주석)은 `RULES.md §5` 참조. 작업 흐름 관점 핵심만:

- 단일 HTML 파일이 기본. 파일이 800줄 넘으면 CSS·JS 분리 검토
- 멀티 파일 프로젝트면 폴더는 **파일 종류가 아니라 기능 단위**로 묶기
- 시맨틱 태그·접근성·디자인 토큰은 `DESIGN.md` 참조
- 사용자가 입력한 HTML 화면 표시 시 `DOMPurify` 등으로 sanitize

## 7. 테스트

기본은 **수동 검증**. 브라우저에서 직접 열어 동작·콘솔 에러 확인.

자동화 테스트가 필요한 경우만:
- **JS 유틸 함수·데이터 변환**: Vitest
- **Python**: pytest
- **브라우저 동작 자동화**: Playwright

원칙:
- 새 의존성 추가는 §3-10 합의 후
- 화면 렌더링 결과는 마크업 단언 대신 **수동/스크린샷**으로 검증
- 타임아웃 기반 단언 금지 (`await sleep(1000)`) → 명시적 대기 조건 사용


## 9. Windows / 한국어 환경 주의

- 경로에 공백·한글 들어가면 따옴표 처리: `"C:\Users\dltmd\OneDrive\Desktop\코드"`
- 파일 인코딩 기본 **UTF-8**. 한국어 깨지면 인코딩 먼저 의심
- Git: `git config --global core.quotepath false` 적용 (한글 파일명 깨짐 방지)
- OneDrive 동기화 폴더에서 작업 중: 큰 빌드 결과물(`node_modules`, `dist`)은 OneDrive 제외 설정 권장
